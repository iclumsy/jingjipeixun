# 小程序学员删除与管理员列表全量加载设计

## 背景

小程序管理员页面当前存在两个使用问题：

1. 学员列表虽已具备后端删除能力和 `canDelete` 能力位，但待审核、已审核、已报名三个状态没有统一可见的删除入口。
2. 学员管理和学习统计都按 20 条分页，并在页面触底时继续加载。当前用户量较小，分批追加会打断浏览，也容易让管理员误以为列表已经完整。

本次改动仅面向小程序管理员页面。普通用户的“我的提交”等列表不调整。

## 目标

- 待审核、已审核、已报名学员均可从统一的“更多操作”弹层执行永久删除。
- 删除只影响本系统的学员记录和关联附件，不调用、不撤销、不修改省网报名。
- 学员管理和学习统计按当前搜索与筛选条件一次加载全部匹配数据。
- 删除、搜索、筛选和下拉刷新后，页面显示完整且不重复的最新结果。

## 非目标

- 不给已驳回、考试通过状态新增删除入口。
- 不修改后端现有删除语义，不新增省网撤销流程。
- 不移除普通用户列表或其他页面的分页能力。
- 不在前端手工维护删除后的统计数字，统一以重新查询结果为准。

## 删除入口设计

### 入口

- 待审核、已审核、已报名卡片均显示“更多操作”按钮。
- 已审核沿用现有按钮；待审核和已报名补充同一按钮。
- “更多操作”弹层根据现有能力位展示操作，并在底部展示红色“删除学员”。
- 删除项仅在 `actions.canDelete` 为真且状态属于 `unreviewed`、`reviewed`、`registered` 时显示。

### 确认与提示

点击删除后关闭“更多操作”弹层，再显示系统确认框：

- 标题：`确认删除学员`
- 通用内容：显示学员姓名，并说明“将永久删除本系统中的学员记录和关联附件，删除后不可恢复”。
- 已报名附加内容：`此操作不会撤销或修改省网报名数据。`
- 确认按钮使用危险操作文案；取消时不发送任何请求。

### 请求与状态

- 确认后调用现有 `api.deleteStudent(studentId)`。
- 删除期间使用页面级操作锁，避免重复点击和重复请求。
- 成功后关闭弹层、提示“已删除学员”，重新加载当前学员列表、项目人数和数据看板。
- 失败时保留列表记录，解除操作锁并展示后端错误信息。

后端继续使用现有 `/api/students/<id>/reject` 和 `{ delete: true }` 语义：删除数据库记录以及本地/COS 关联附件，不操作省网。

## 学员管理全量加载

### 页面状态

从管理员学员管理页面移除以下分页状态与行为：

- `page`
- `limit`
- `hasMore`
- 触底调用 `loadRecords(false)`
- 列表追加合并
- “已显示全部记录”提示

### 查询行为

- 首次进入、切换状态、切换项目、搜索、清空搜索、下拉刷新和删除成功后，都重新请求当前条件的完整集合。
- 每次成功响应直接替换 `records`，不与旧列表拼接。
- 项目列表、项目人数和总匹配人数仍来自完整未分页集合。
- 通用 `api.getStudents` 增加明确的全量选项供管理员页面使用；默认分页行为保留，避免影响普通用户页面。

## 学习统计全量加载

### 页面状态

从学习统计分栏移除以下分页状态与行为：

- `reportPage`
- `reportLimit`
- `reportHasMore`
- 触底调用 `loadReportRecords(false)`
- 列表追加合并
- “已显示全部学习记录”提示

### 接口与查询

- 小程序管理员学习统计接口不再接收或使用 `page`、`limit`。
- 状态、项目和搜索条件应用完成后，返回完整 `filtered_list`。
- 学习时长计算应用于所有返回记录，不再只计算当前分页切片。
- 响应保留 `total`，移除页面不再使用的分页字段。
- 页面每次成功响应直接替换 `reportList`。
- 现有请求序号保护继续保留，防止快速切换筛选时旧响应覆盖新结果。

## 数据流

### 删除

1. 管理员在卡片点击“更多操作”。
2. 页面保存当前学员并打开操作弹层。
3. 管理员点击“删除学员”。
4. 页面展示包含姓名和删除范围的二次确认。
5. 确认后调用现有删除 API。
6. 成功后完整刷新学员管理列表及统计；失败时保留当前数据并提示错误。

### 查询

1. 用户进入页面或改变筛选条件。
2. 页面发送一次带搜索、状态、项目条件的请求，不发送分页参数。
3. 服务端返回全部匹配结果和聚合信息。
4. 页面用响应完整替换当前列表。
5. 页面触底不执行任何网络请求。

## 错误处理

- 查询失败时保留已有列表，结束加载态并提示“加载失败”。
- 删除失败时不从前端列表预先移除记录。
- 删除过程中忽略新的删除触发，保证单次操作只有一个在途请求。
- 学习统计继续使用请求序号校验，只接收最近一次筛选请求的响应。

## 测试策略

### 前端删除行为

- 待审核、已审核、已报名均显示“更多操作”。
- 三个目标状态的弹层均显示删除项；非目标状态不新增入口。
- 取消确认不调用 API。
- 确认后以正确学员 ID 调用 `api.deleteStudent`。
- 已报名确认文案明确说明不影响省网报名。
- 删除成功触发完整刷新；删除失败保留记录并解除操作锁。

### 前端全量加载

- 学员管理请求使用全量模式，响应直接替换列表。
- 学习统计请求不发送分页参数，响应直接替换列表。
- 页面触底在两个分栏中都不发起加载请求。
- 搜索、状态筛选、项目筛选和下拉刷新不会产生重复记录。
- WXML 不再显示分页结束提示。

### 后端学习统计

- 接口返回当前搜索、状态和项目条件下的全部记录。
- `total` 与返回列表长度一致。
- 每条返回记录都包含学习时长字段。
- 现有权限校验、状态过滤和项目统计保持不变。

## 验收标准

- 管理员可在待审核、已审核、已报名卡片的“更多操作”中删除学员。
- 已报名删除确认明确说明省网数据不受影响。
- 删除成功后学员从当前完整列表消失，筛选人数和看板同步更新。
- 学员管理和学习统计首次加载即显示全部匹配记录。
- 滚动到底部不再出现加载动作或分页结束提示。
- 普通用户列表行为不发生变化。
